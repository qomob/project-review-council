# Example: GitHub 开源项目评审（Few-shot 标杆）

> 场景：开源开发者工具，GitHub Star 8K，无商业模式，维护者 2 人。
> 此示例展示非商业项目的评审适配——商业模式维度降权，社区/生态维度加权。

## 配置调整（非商业项目适配）

```yaml
# 此类项目可临时调 weights.yaml
dimensions:
  commercial_value:    { weight: 0.05 }   # 降权
  business_model:      { weight: 0.02 }   # 降权
  profitability:       { weight: 0.02 }   # 降权
  moat:                { weight: 0.20 }   # 加权（社区壁垒）
  user_value:          { weight: 0.20 }   # 加权
  innovation:          { weight: 0.15 }   # 加权
```

## P1 项目画像摘要

- **项目目标**：CLI 工具，自动化代码审查
- **目标用户**：独立开发者 / 小团队
- **商业模式**：无（纯开源）
- **核心价值**：免费 + 本地运行 + 隐私友好
- **当前阶段**：成熟期（8K Star，200+ 贡献者）

## P5 风险报告摘要

### Fatal（1 项）

| # | 风险 | 影响范围 | 概率 | 解决成本 | 优先级 |
|---|------|---------|------|---------|--------|
| 1 | 维护者仅 2 人，bus factor=2 | 项目存续 | 中 | M | P0 |

### High（2 项）

| # | 风险 | 影响范围 | 概率 | 解决成本 | 优先级 |
|---|------|---------|------|---------|--------|
| 1 | 无收入，维护者倦怠风险高 | 停更 | 高 | L | P1 |
| 2 | 无商业模式 = 无可持续路径 | 长期存活 | 高 | L | P1 |

## P7 商业评估（适配后）

| 维度 | 评分 | 理由 |
|------|------|------|
| 商业价值 | 2 | 无收入【证据】 |
| 技术价值 | 8 | 架构清晰【证据】 |
| 用户价值 | 9 | 8K Star 解决真痛点【证据】 |
| 市场价值 | 5 | 市场大但免费【推测】 |
| 创新程度 | 7 | 本地隐私审查有差异【证据】 |
| 竞争壁垒 | 6 | 社区 + 贡献者生态【证据】 |
| 可复制性 | 7 | 代码可 fork 但社区难复制【证据】 |
| 商业模式 | 1 | 无【证据】 |
| 盈利能力 | 1 | 无【证据】 |
| 增长潜力 | 5 | Star 增长但无变现【证据】 |
| 全球化潜力 | 8 | 无地域依赖【证据】 |

**加权综合分**：5.5/10
**对应评级**：C+（修正后 B-，因社区壁垒加权）
**建议**：继续（开源），但需建立可持续维护机制

## P9 决议关键判断

**如果这是你的项目**：会继续做，但第一件事是建立可持续模型（GitHub Sponsors / Open Collective / 付费企业版），否则倦怠是必然结局。

---

**用法**：此示例展示非商业/开源项目的适配方式——通过调 `weights.yaml` 而非改 Prompt。
